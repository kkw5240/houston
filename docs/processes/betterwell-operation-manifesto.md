# My Project Operation Manifesto: Sprint & QA Strategy

> **버전:** 1.1 (2026-03-05 회고 반영)
> **대상 프로젝트:** My Project (my-project)
> **목적:** AI를 활용한 초고속 개발 속도와 제품의 '품질 안정성' 간의 균형을 맞추기 위한 스프린트/배포/QA 운영 원칙 (Ground Rules).
> **AI 프롬프트 가이드:** 다른 AI 에이전트에게 회고(Retrospective)를 요청할 때, 이 문서 전체를 맥락으로 제공하여 현재 우리 팀의 규칙을 학습시키세요.

---

## 1. 팀 환경 및 구성 (Team Context)

현재 우리 팀은 빠른 속도와 다중 프로젝트 병렬 진행이 특징인 목적 조직(Squad)입니다.
*   **PM (겸 Backend 개발):** 전체 스프린트 타임라인 설계, 리스크 방어, 주력 백엔드 서버 로직 개발
*   **PO (겸 QA - 신입):** 비즈니스 요구사항 정의, Sprint Goal 초안 작성, 스프린트 배포 전 기능 QA(사용성 테스트) 전담, **이슈 등록 시 QA TC(Test Case) 작성**
*   **Dev Team (FE, MB, Design):** 각 파트별 UI/UX 및 클라이언트 개발

**핵심 개발 환경 (AI-Assisted Development):**
*   모든 개발자(BE, FE, MB)는 Claude Code, Gemini 등의 AI 에이전트를 적극 활용합니다.
*   이를 통해 **기능 구현(코딩) 속도는 비약적으로 빠르나, 전체 아키텍처나 부수 효과(Side-effect)에 대한 100% 이해도(Context)가 부족하여 회귀 이슈가 발생하기 쉬운 환경**입니다.

## 2. 현재의 문제점 (As-Is Pain Points)

이 선언문이 만들어지게 된 근본 원인(안정화가 시급한 이유)입니다.
1.  **쏟아지는 긴급 요구사항:** 기획/영업 등 외부 파트의 지속적인 우선순위 개입("이게 제일 중요해요")으로 기존 계획 상실
2.  **QA 병목 & 회귀 버그 폭발:** AI를 통한 빠른 개발 ➡️ 충분한 회귀 테스트 생략 ➡️ 스테이지(Stage) 투입 ➡️ QA/PM 과부하 및 실제 운영 서버 오류 발생 (Ex. Payroll 한도수당 중복합산, 퇴사처리 권한 꼬임 등)
3.  **지친 금요일:** 스프린트 종료일(금요일)에 배포와 마일스톤 회고/플래닝이 겹쳐 집중도 저하

## 3. 새로운 스프린트 운영 정책 (To-Be Strategy)

위 문제를 해결하기 위해, 우리는 다음과 같이 **품질 우선 스케줄(Quality-First Schedule)**을 표준으로 채택합니다.

### 3.1. 버퍼 확보 룰 (Capacity Allocation)

멀티 프로젝트를 감안하여, **파트별로 스프린트 할당 가능 시간을 산정**합니다.

| 구분 | BE (PM 겸) | FE | MB |
|:---|:---:|:---:|:---:|
| 주 가용 시간 | 40h | 40h | 40h |
| PM/관리 오버헤드 | -8h | — | — |
| Hotfix/긴급 버퍼 | -4h | -2h | -2h |
| 회의/리뷰/소통 | -4h | -4h | -4h |
| **할당 가능** | **24h (60%)** | **34h (85%)** | **34h (85%)** |

*   남은 시간은 **'안전 버퍼(Safety Buffer)'**로 철저히 남겨둡니다.
    *   AI 코드 결과물의 엣지 케이스 및 부수 효과 분석 (회귀 방어)
    *   기획/영업의 초긴급(Hotfix) 요구사항 대응 공간
    *   코드 리뷰 및 로컬 환경 - 검증 서버 간의 동시성/인프라 오차 확인

#### Velocity 측정 — "Yesterday's Weather" 방식

위 할당 시간은 초기 기준값이며, **개인별 실적 데이터로 교정**합니다.

1.  매 Sprint 금요일에 **"계획 건수 vs 완료 건수"를 파트별·개인별로 기록** (5분)
2.  3 Sprint 후 평균 완료율을 보고 각 파트의 할당 가능 시간을 교정
3.  **Hotfix 건수도 별도 기록** — 긴급건이 주 2건 이상이면 긴급 버퍼를 올릴 것

```
예시)
Sprint 28:  PM/BE 계획 8건 → 완료 6건 → 완료율 75%  → 다음 Sprint에 6건 할당
Sprint 29:  PM/BE 계획 6건 → 완료 6건 → 완료율 100% → 안정 ✅
```

> 🤖 **For AI Agents:** "이번 주 Velocity를 알려줘"라고 요청 받으면, 해당 Sprint의 계획 건수/완료 건수/완료율을 파트별로 집계하여 보고하세요.

### 3.2. 일방향 배포 스케줄 (Weekly Flow)

개발 완료와 QA, 그리고 배포를 철저히 분리합니다.

*   **월 ~ 화 (집중 개발기간):** TDD/BDD 규칙을 철저히 지키며 기능 개발 및 버그 핫픽스 집중. (1순위 이슈 우선 처리)
    *   *AI Agent 가이드:* 이 기간엔 기능 추가(Feat)에 집중하되 테스트 코드를 반드시 짜도록 강제할 것.
*   **수 (QA 인계 마감선 & 배포 범위 확정):** 수요일 오전 10시까지 `🕵️ QA` 라벨이 붙지 않은 이슈는 **이번 배포에서 제외**, 다음 배포로 자동 이월. 개발자가 직접 작성한 기능을 검증하고 PO에게 QA 인계.
    *   코드 작성을 멈추는 것이 아니라, **이번 배포에 들어갈 범위를 확정**하는 것.
    *   수요일 이후 발견된 버그 핫픽스는 PM 판단 하에 배포 포함 가능.
*   **목 (검증 및 정기 배포):** QA 완료된 건에 한해 **오후 정기 배포**. 배포 전 **Release Plan(배포계획서)** 작성 필수. 배포 후 모니터링 즉시 착수.
*   **금 (스프린트 Wrap-up Day):** PM은 신규 이슈 할당 지양, 회고/플래닝에 집중하는 날.
    *   [오전] 개인별 GitHub 이슈 업데이트, 다음 주 할당될 이슈별 예상 공수(Hours) 산출, Velocity 기록
    *   [오후] **Kudos (5분)** ➡️ 스프린트 리뷰 ➡️ 회고(KPT) ➡️ 플래닝
        *   **Kudos:** 각자 이번 주 도움 받은 사람 또는 인상적이었던 기여를 한 줄로 공유
        *   **플래닝:** PO가 작성한 Sprint Goal 초안을 PM과 함께 확정 → 다음 주 티켓 점검 → Velocity 기반 할당 시간 커트라인 확정

### 3.3. 투명한 마일스톤 공유 (SLA Management)

*   금요일 플래닝에서 합의된 티켓 이외에 무조건 밀어넣는 긴급 요청은 기획/영업이 우선순위를 결정(기존 1순위를 포기할지)하게 만듭니다.
*   스프린트 미팅 날, 다가올 배포(다음 주 목요일)에 나갈 버전의 "릴리즈 예정 마일스톤 노트"를 작성하여 영업채널에 명시합니다.

## 4. 팀 그라운드 룰 (Rules of Engagement)

아래의 룰은 직책을 막론하고 무조건 준수합니다.

1.  **"No Issue, No Work":** 구두 연락(전화, 지나가다 한마디 등)으로 작업하지 않습니다. 기획/영업의 모든 문의 사항은 PO(QA)의 통제 하에 "GitHub 이슈"로 등록되어야만 백로그 테이블에 올라갈 수 있습니다.
    *   **예외 — 자율 개선 이슈:** 개발자가 발견한 기술 부채나 DX 개선은 `tech-debt` 또는 `dx-improvement` 라벨을 붙여 **PO 게이트 없이** 직접 이슈 등록 가능.
2.  **"Developer's Check First":** 개발의 책임은 AI나 PO가 지지 않습니다. 자신이 짠 코드(AI가 짠 부분 포함)의 로컬 동작 검증(Integration Test)을 거치지 않은 이슈는 `🕵️ QA` 라벨을 달아 PO에게 던지지 마십시오.
3.  **"Focus on Regressions":** 로직을 짤 때마다 **"내가 건드린 테이블을 의존하는 다른 낡은 모듈(Legacy)이 터지지 않을까?"**를 항상 AI에게 물어보고 사이드 이펙트를 기록에 남기십시오.
4.  **"QA = TC 기반 검증":** PO는 이슈 등록 시 **QA TC(Test Case)**를 함께 작성합니다. 배포 대상 조건: PO가 작성한 TC가 존재하고, **전건 PASS**일 때만 해당 이슈가 배포에 포함됩니다.

## 5. 긴급 대응 SLA (Incident & Priority)

| 등급 | 대응 시간 | 해결 목표 | 채널 | 비고 |
|:---|:---|:---|:---|:---|
| 🔴 **장애 (Outage)** | 즉시 | 30분~1시간 | Slack `#긴급장애대응` | 실시간 상황 보고: `[원인] → [대응 중] → [ETA] → [해결]` |
| **P0** (Critical) | 당일 | 이번 주 내 | GitHub Issue | 접수일 기준 |
| **P1** (High) | 2영업일 | 다음 주 내 | GitHub Issue | 접수일 기준 |
| **P2** (Normal) | 1 Sprint | 다다음 주 | GitHub Issue | 접수일 기준 |

*   장애 등급(Outage)의 경우, AI Agent가 처리 중간중간 Slack에 **원인 분석 → 해결 방안 → 예상 소요 시간**을 보고합니다.
*   금요일 플래닝에서 합의된 티켓 이외의 긴급 요청은 **기존 P0 이슈 1건을 버릴 것인지** 기획/영업이 결정하게 만듭니다.

## 6. Sprint Goal & Velocity 추적

### Sprint Goal
*   **PO가 초안 작성** ("이번 주 고객/사업 관점에서 가장 중요한 것" 1~2줄)
*   PM이 기술적 맥락을 보완
*   금요일 플래닝에서 **팀 합의로 확정**
*   Sprint 내 의사결정에서 Sprint Goal에 부합하는 이슈를 우선

### Velocity 기록 양식 (금요일 기록)

```
Sprint {N} Velocity (YYYY-MM-DD)
---
[BE/PM] 계획: {X}건 / 완료: {Y}건 / 완료율: {Y/X}% / Hotfix: {Z}건
[FE]    계획: {X}건 / 완료: {Y}건 / 완료율: {Y/X}%
[MB]    계획: {X}건 / 완료: {Y}건 / 완료율: {Y/X}%
---
다음 Sprint 할당: [BE] {Y}건 / [FE] {Y}건 / [MB] {Y}건
```

---

> 🤖 **For AI Agents:** 만약 당신이 이 문서를 읽었다면, 사용자가 "이번 주 스프린트 회고를 작성해줘"라고 명령할 때, 다음을 중점적으로 점검하여 날카로운 칭찬과 비판을 해주세요:
> 1. **Velocity 기반 할당 룰** — 파트별 할당 시간 대비 실제 처리량
> 2. **목요일 정기 배포 룰** — 배포 스케줄 준수 여부
> 3. **TDD/BDD 룰** — 테스트 코드 동반 여부
> 4. **수요일 QA 인계 마감선** — `🕵️ QA` 라벨 부착 기한 준수
> 5. **QA TC 기반 검증** — TC 존재 및 전건 PASS 여부
> 6. **Sprint Goal 달성 여부** — 이번 Sprint의 핵심 목표 달성 평가
